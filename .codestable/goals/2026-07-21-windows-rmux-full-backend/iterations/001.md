# Iteration 001

## Current Understanding

`rmux 0.9.0` 可作为 Windows native backend 的候选基座，但必须把 tmux runner 的默认 capture 语义拆开处理：

- lifecycle 命令 `start-server` / `new-session` 在 Windows 下不能走 PIPE capture，否则会挂起。
- foreground attach 必须继承终端 stdio，非 TTY probe 不能直接证明 attach 语义。
- Windows logical EOF 对 rmux 需要发送 `C-z Enter`，不能只发送 `C-d`。
- `capture-pane -S -N` 与 tmux last-N 行语义不等价，需要 client-side 非空尾行裁剪。
- provider pane status parser 需要处理 OSC 等非 CSI terminal control sequence，避免 rmux/tmux capture 下状态误判。

## Implementation Approach

选择真实 backend/runner 分层，而不是在 gate 中 fake pass：

- 新增 rmux 专用 runner，集中封装 stdio-aware subprocess 策略和 logical key 映射。
- 新增 `RmuxBackend` 复用 `PsmuxBackend` 的 mux API 形状，但覆盖 rmux 差异点。
- foreground attach 按 ping payload 的 `namespace_backend_impl` 分支到 rmux，attach 进程不设置 stdout/stderr。
- gate 增强为“非交互 full probe + live foreground/client evidence”组合，保留原始失败 artifact，并显式记录 accepted workaround。

## Problems Encountered

- 直接在非 TTY probe 中运行 `attach-session` 会输出 `open terminal failed: not a terminal`，因此不能把该场景当作 rmux attach 缺能力；需要 live foreground evidence。
- `refresh-client -t <session>` 在无 live client 时返回 `can't find client`，实际语义需要用 `list-clients` 观察到的 client id 验证。
- main probe 的 `kill-server` 发生在 `kill-session` cleanup 后，此时 server 已不存在；需要 live server 场景单独证明。
- 全量 `python -m pytest -q` 在收集阶段缺少 `jsonschema`，排除这两个文件后的广义测试在 240 秒超时；本轮没有安装依赖或修改全局环境。
