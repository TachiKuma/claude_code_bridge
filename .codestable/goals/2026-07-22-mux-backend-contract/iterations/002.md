# Iteration 002

## Current Understanding

owner 已解除 iteration 001 的 owner-stop：`backend-resolver-opt-in-contract` 已验收通过，继续推进 `mux-backend-contract`。

CMD-004 的 Windows 黑盒失败不再是 token ACL 问题。新的根因是当前环境的 `tmux` 实际解析到 `C:\Users\Administrator\AppData\Local\psmux\tmux.EXE`，psmux 兼容层会在部分 tmux UI/policy 命令后让 session 消失。

## Implementation Approach

本轮选择 capability gate，而不是为 psmux 伪造完整 tmux 行为。`ensure_server_policy` 保留 namespace 生命周期必需的基础 server option；对 psmux/rmux 兼容层跳过 environment sync、window option 和 keybinding。`apply_project_tmux_ui` 在兼容层下整体 no-op，因为 `status-style` 等主题命令也会破坏 session。

window materialization 改为优先使用 `list-windows` 返回的 window id 作为后续 target，避免依赖 window name target 的实现细节。

## Changes This Iteration

- 新增 `terminal_runtime.tmux_compat.is_tmux_compat_subset`，统一识别 `backend_impl=psmux/rmux` 或 PATH 中解析到 psmux/rmux 的 tmux 兼容层。
- `ensure_server_policy` 在兼容层下跳过环境同步、window policy 和 keybindings。
- `apply_project_tmux_ui` 在兼容层下不发送项目 UI/主题命令。
- `materialize_topology` 在确保窗口后记录 `window_id`，后续 root pane 解析和 entry window selection 优先使用 window id。
- `_run_ccb` 测试 helper 固定 `encoding='utf-8', errors='replace'`，避免 Windows GBK 解码线程异常遮蔽真实 stderr。
- 增加 psmux 兼容层 policy/UI regression tests。
