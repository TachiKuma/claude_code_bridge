# 陷阱研究：i18n 与多 AI 协作

**领域:** i18n 国际化 + 多 AI 协作系统
**研究日期:** 2026-03-28
**置信度:** HIGH

## 关键陷阱

### 陷阱 1: 协议字符串被错误翻译

**出错表现:**
将系统协议标记（如 `CCB_DONE`、`ask.response`、环境变量名、JSON 键）当作用户文本进行翻译，导致跨进程通信完全失败。

**发生原因:**
- 开发者未区分"人类可读文本"和"机器协议字符串"
- i18n 工具自动提取所有字符串，包括协议标记
- 翻译人员缺乏技术上下文，将所有内容都翻译了

**如何避免:**
- 在 i18n 架构中明确区分两类字符串：`HUMAN_TEXT` vs `PROTOCOL_CONSTANTS`
- 协议标记使用常量定义，永不通过 `t()` 函数包装
- 添加 CI 检查：扫描代码确保协议字符串未被 `t()` 包装
- 为翻译人员提供明确的"不可翻译列表"文档

**警告信号:**
- 异步请求突然全部失败，日志显示无法识别的响应类型
- 守护进程无法解析来自适配器的消息
- 环境变量读取返回 `None`（因为变量名被翻译了）
- 会话文件 JSON 解析失败

**应在阶段解决:**
Phase 1（架构设计阶段）- 在 i18n_core 设计时就建立协议/文本分离机制

---

### 陷阱 2: 多 AI 协作中的上下文崩溃

**出错表现:**
当任务在多个 AI 提供商之间传递时，关键信息逐渐丢失或扭曲，导致后续 AI 产生与原始目标矛盾的输出。

**发生原因:**
- 每次 AI 切换都会压缩/重新解释上下文
- 会话历史超过上下文窗口限制时，早期指令被截断
- 不同提供商对相同提示的理解存在差异
- 缺乏显式的"任务状态"传递机制

**如何避免:**
- 实现结构化的 `TaskHandle` 对象，包含：
  - 原始目标（不可变）
  - 当前状态（明确的枚举值）
  - 已完成的子任务列表
  - 待处理的依赖项
- 在每次 AI 切换时，显式传递完整的任务上下文
- 使用会话文件持久化关键决策和中间结果
- 限制协作链深度（建议不超过 3 层）

**警告信号:**
- AI 重复执行已完成的任务
- 后续 AI 的输出与前面 AI 的决策矛盾
- 任务在两个 AI 之间来回传递但无进展（"乒乓循环"）
- 日志显示 AI 询问已经回答过的问题

**应在阶段解决:**
Phase 2（集成设计阶段）- 设计 CCBCLIBackend 时实现 TaskHandle 机制

### 陷阱 3: 硬编码字符串散落导致翻译不完整

**出错表现:**
应用看起来已国际化，但实际使用时仍有大量英文文本出现，用户体验支离破碎。

**发生原因:**
- 1000+ 处硬编码字符串分散在 98 个文件中
- 动态生成的错误消息未被提取
- 日志消息被认为"不需要翻译"而跳过
- 命令行帮助文本、异常消息被遗漏

**如何避免:**
- 使用自动化工具扫描代码库，标记所有未包装的字符串字面量
- 建立"翻译覆盖率"指标，纳入 CI 检查
- 区分三类文本：
  - 用户界面文本（必须翻译）
  - 日志/调试信息（可选翻译）
  - 协议字符串（永不翻译）
- 采用渐进式迁移策略：按模块优先级逐步国际化

**警告信号:**
- 用户报告"部分中文，部分英文"
- 错误消息始终是英文
- 命令行 `--help` 输出未翻译
- 翻译文件中的键数量远少于代码中的字符串数量

**应在阶段解决:**
Phase 1（架构设计阶段）- 建立扫描工具和覆盖率指标
Phase 3（实施阶段）- 按模块逐步迁移

### 陷阱 4: 多 AI 协作的"协调税"爆炸

**出错表现:**
添加更多 AI 提供商后，系统性能急剧下降，成本飙升，但输出质量并未提升。

**发生原因:**
- 每次 AI 切换都需要完整的上下文传递（token 成本）
- 验证循环和"审查抖动"消耗大量资源
- 协调开销随 AI 数量呈指数增长
- 缺乏明确的任务分配策略，导致重复工作

**如何避免:**
- 为每个 AI 提供商定义明确的专长领域和触发条件
- 实现"模型路由"：简单任务用小模型，复杂推理用大模型
- 限制并行 AI 数量（CCB 当前 4 个提供商是合理上限）
- 添加监控：跟踪每个提供商的 token 消耗和响应时间
- 避免"全员参与"模式，使用"按需调用"

**警告信号:**
- API 成本突然增加 3-5 倍
- 任务完成时间随 AI 数量增加而延长（而非缩短）
- 日志显示多个 AI 在做相同的分析工作
- 大量时间花在"规划"而非"执行"

**应在阶段解决:**
Phase 2（集成设计阶段）- 设计路由策略和监控机制
Phase 4（优化阶段）- 根据监控数据调整策略

### 陷阱 5: 文本扩展破坏 UI 布局

**出错表现:**
切换到德语、芬兰语等语言后，按钮文本被截断，对话框溢出，界面元素重叠。

**发生原因:**
- 假设所有语言的文本长度与英文相似
- 使用固定宽度容器或硬编码像素值
- 未考虑某些语言（德语、芬兰语）比英文长 30-50%

**如何避免:**
- 使用弹性布局（flexbox、grid），避免固定宽度
- 在设计阶段使用伪本地化测试（用加长的假文本）
- 为 CLI 输出使用动态列宽计算
- 预留 30-50% 的额外空间用于文本扩展

**警告信号:**
- 非英语用户报告界面"看起来很乱"
- 按钮文本显示为 "Speiche..."（被截断）
- 表格列对齐错乱
- 帮助文本换行异常

**应在阶段解决:**
Phase 1（架构设计阶段）- 建立布局原则
Phase 3（实施阶段）- 伪本地化测试

### 陷阱 6: 会话管理在多 AI 环境中的竞态条件

**出错表现:**
并发调用多个 AI 提供商时，会话文件损坏，JSON 解析失败，请求丢失或重复。

**发生原因:**
- 多个适配器同时读写同一会话文件
- 缺乏文件锁定机制或锁定实现不完整
- 当前的"重试 10 次"策略只是掩盖问题，未根本解决
- 跨进程锁定在高并发下失效

**如何避免:**
- 实现原子性会话写入（写入临时文件 → 原子重命名）
- 使用进程级文件锁（`fcntl.flock` 或 `msvcrt.locking`）
- 为每个提供商使用独立的会话目录，减少冲突
- 添加会话版本号和校验和，检测损坏
- 实现会话恢复机制，而非简单重试

**警告信号:**
- 日志中频繁出现 JSON 解析错误
- 会话文件内容为空或截断
- 第二个异步请求总是失败（OpenCode 死锁问题的变体）
- 重试延迟累积导致响应时间达到 500ms+

**应在阶段解决:**
Phase 2（集成设计阶段）- 重新设计会话管理机制
Phase 3（实施阶段）- 添加原子写入和锁定

### 陷阱 7: 字符串拼接破坏翻译

**出错表现:**
翻译后的句子语法错误、词序混乱，或完全无法理解。

**发生原因:**
- 使用代码拼接句子片段：`_("You have") + " " + str(count) + " " + _("messages")`
- 不同语言的语法结构、词序、性别变化差异巨大
- 翻译人员只看到片段，无法理解完整上下文

**如何避免:**
- 永远翻译完整句子，使用占位符：`_("You have {count} messages").format(count=count)`
- 对于复数形式，使用 `ngettext()`：
  ```python
  ngettext("You have {n} message", "You have {n} messages", count).format(n=count)
  ```
- 避免在代码中构建句子结构
- 为翻译人员提供完整的句子上下文和截图

**警告信号:**
- 翻译后的文本读起来像"机器翻译"
- 词序明显错误（如中文出现"消息 3 你有"）
- 复数形式在所有数量下都相同
- 翻译人员反馈"无法理解这个片段的含义"

**应在阶段解决:**
Phase 1（架构设计阶段）- 建立翻译规范和代码审查规则
Phase 3（实施阶段）- 重构现有拼接代码

### 陷阱 8: 多 AI 协作中的隐式假设

**出错表现:**
AI 提供商之间传递任务时，接收方误解任务意图，产生不相关或错误的输出。

**发生原因:**
- 依赖 AI"推断"任务要求，而非显式定义
- 缺乏类型化的接口和 schema
- 不同 AI 模型对相同自然语言提示的理解存在差异
- 任务边界和职责划分模糊

**如何避免:**
- 为每个 AI 角色定义显式的"行动 schema"：
  - 输入格式（JSON schema）
  - 预期输出格式
  - 允许的操作列表
  - 成功/失败标准
- 使用结构化数据而非自然语言传递任务
- 实现任务验证层，检查输入/输出是否符合 schema
- 为每个角色编写明确的系统提示，定义边界

**警告信号:**
- AI 执行了未被要求的操作
- 输出格式与预期不符，需要大量后处理
- 任务在 AI 之间"踢皮球"，无人承担责任
- 相同输入产生高度不一致的输出

**应在阶段解决:**
Phase 2（集成设计阶段）- 定义角色 schema 和接口契约
Phase 3（实施阶段）- 实现验证层

## 技术债务模式

| 捷径 | 即时收益 | 长期成本 | 可接受条件 |
|------|---------|---------|-----------|
| 跳过伪本地化测试 | 节省 1-2 天测试时间 | UI 在非英语环境下完全破碎，需要大规模返工 | 永不可接受 |
| 使用全局 `_()` 函数而非命名空间 | 代码更简洁 | 多个库冲突，无法区分翻译来源 | 仅限单一应用（非库） |
| 依赖空闲超时检测完成 | 避免修改提供商提示 | 误判完成导致输出截断，或长时间挂起 | 仅作为后备机制，必须有主要检测 |
| 手动管理 PO/POT 文件 | 无需配置自动化工具 | 翻译遗漏、版本不同步、人为错误频发 | 永不可接受（必须自动化） |
| 所有任务都用最大模型 | 输出质量最高 | 成本爆炸，响应时间慢 | 仅限原型阶段 |
| 使用轮询而非文件锁 | 实现简单 | 性能差（500ms 延迟），资源浪费 | 仅限低并发场景（<5 req/min） |

## 集成陷阱

| 集成点 | 常见错误 | 正确方法 |
|--------|---------|---------|
| CCB CLI 调用 | 解析控制台文本输出（易碎） | 使用结构化响应文件（JSON） |
| 会话文件共享 | 多进程直接读写同一文件 | 使用文件锁 + 原子写入 |
| 语言检测 | 硬编码 `CCB_LANG` 环境变量 | 支持多种检测方式（环境变量、配置文件、命令行参数） |
| AI 提供商切换 | 重新发送完整历史记录 | 使用会话 ID 引用，仅传递增量 |
| 翻译键命名 | 使用通用键如 `error_1` | 使用层级化、描述性键如 `ccb.adapter.gemini.timeout_error` |
| 完成标记检测 | 仅依赖 `CCB_DONE` 字符串 | 多重检测：标记 + 超时 + 内容模式 |

## 性能陷阱

| 陷阱 | 症状 | 预防措施 | 何时崩溃 |
|------|------|---------|---------|
| 会话 JSON 重试轮询 | 响应延迟 50-500ms | 使用异步文件锁或文件监视器 | 并发请求 >10/秒 |
| 上下文窗口溢出 | AI 忘记早期指令 | 实现上下文压缩和关键信息提取 | 会话消息 >50 条 |
| 翻译文件线性查找 | 启动时间随翻译数量增长 | 使用字典缓存，延迟加载 | 翻译键 >1000 个 |
| 所有 AI 串行调用 | 总时间 = 各 AI 时间之和 | 并行调用独立任务 | 协作链 >3 层 |
| 未限制工作池大小 | 内存耗尽，系统崩溃 | 配置最大并发数，实现队列 | 并发任务 >20 个 |

## 安全错误

| 错误 | 风险 | 预防措施 |
|------|------|---------|
| 翻译文件中包含 API 密钥 | 密钥泄露到翻译服务或版本控制 | 永不翻译配置值，仅翻译用户文本 |
| 未验证 AI 输出就执行 | AI 生成恶意命令或代码 | 实现输出验证和沙箱执行 |
| 会话文件权限过宽 | 其他用户可读取敏感对话 | 设置严格的文件权限（600） |
| 日志记录用户输入 | 敏感信息泄露到日志文件 | 实现日志脱敏，标记敏感字段 |
| 跨进程锁定失败 | 竞态条件导致数据损坏 | 实现锁定超时和死锁检测 |

## UX 陷阱

| 陷阱 | 用户影响 | 更好的方法 |
|------|---------|-----------|
| 混合语言输出 | 困惑、不专业的印象 | 确保 100% 翻译覆盖率 |
| 未翻译错误消息 | 用户无法理解问题，无法自助解决 | 优先翻译所有错误和帮助文本 |
| 多 AI 协作对用户不透明 | 用户不知道为何等待，感觉系统卡住 | 显示进度："正在咨询 Codex 进行代码审查..." |
| 语言切换需要重启 | 用户体验中断 | 支持运行时语言切换 |
| 翻译质量低劣 | 用户宁愿用英文 | 提供上下文给翻译人员，使用专业翻译服务 |

## "看起来完成但实际未完成"检查清单

- [ ] **i18n 覆盖率**: 已扫描所有模块，翻译覆盖率 >95% — 验证：运行自动化扫描工具
- [ ] **协议字符串隔离**: 所有协议标记未被 `t()` 包装 — 验证：CI 检查通过
- [ ] **伪本地化测试**: UI 在文本扩展 50% 时仍正常 — 验证：运行伪本地化测试套件
- [ ] **多 AI 并发测试**: 3 个提供商并发调用无竞态条件 — 验证：压力测试 100 次无失败
- [ ] **会话文件原子性**: 并发写入不会损坏文件 — 验证：混沌测试（模拟进程崩溃）
- [ ] **上下文传递完整性**: 任务在 AI 间传递时信息无丢失 — 验证：端到端集成测试
- [ ] **翻译质量审查**: 母语者审查所有翻译 — 验证：人工审查报告
- [ ] **性能基准**: 多语言环境下性能无明显下降 — 验证：基准测试对比
- [ ] **错误恢复**: 所有失败场景都有恢复路径 — 验证：故障注入测试
- [ ] **监控和可观测性**: 可追踪每个 AI 的 token 消耗和决策 — 验证：查看监控仪表板
