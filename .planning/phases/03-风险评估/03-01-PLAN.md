**报告结构:**

```markdown
# 协议字符串误翻译风险评估报告

**评估日期:** 2026-03-28
**Phase:** 03-风险评估
**需求:** RISK-01

## 1. 执行摘要

- **严重程度:** 完全破坏（Critical）
- **影响范围:** CCB 守护进程通信、环境变量、文件路径
- **缓解策略:** 双层保护（CI 检查 + 运行时验证）
- **残留风险:** 低（白名单维护疏漏）

## 2. 风险影响评估

### 2.1 严重程度分级

**完全破坏（Critical）** - 协议字符串被翻译后，CCB 系统完全无法工作

### 2.2 影响场景

**场景 1: 完成标记被翻译**
- 问题: ask.response 被翻译为 "询问.响应"
- 影响: 守护进程无法识别回复标记
- 后果: 跨进程通信失败

**场景 2: 环境变量名被翻译**
- 问题: CCB_LANG 被翻译为 "CCB_语言"
- 影响: 无法读取环境变量
- 后果: 语言检测失败，回退到默认值

**场景 3: 文件路径被翻译**
- 问题: .planning 被翻译为 ".规划"
- 影响: 文件路径错误
- 后果: 无法读取规划文件

### 2.3 量化影响

基于 Phase 1 数据:
- 协议字符串总数: {从 protocol_whitelist.json 读取 total_count}
- 高风险类别: 环境变量（{env_vars 数量}）、命令名（{command_names 数量}）、完成标记（{completion_markers 数量}）
- 中风险类别: 文件路径（{file_paths 数量}）、JSON 键（{json_keys 数量}）
- 低风险类别: Git 引用（{git_refs 数量}）、配置键（{config_keys 数量}）

## 3. 风险来源分析

### 3.1 CI 检查漏洞（最高风险）

**问题:** 外部翻译文件（~/.ccb/i18n/）不在版本控制中

**风险等级:** 高

**发生概率:** 中等（用户可能创建自定义翻译）

**缓解策略:** 运行时验证（per D-03）

### 3.2 开发者误用

**问题:** 误将协议字符串包装在 t() 函数中

**示例:**
```python
# ❌ 错误
command = t("ccb.command.ask")  # 返回 "ask"

# ✓ 正确
ASK_COMMAND = "ask"
print(t("ccb.help.ask_usage", cmd=ASK_COMMAND))
```

**风险等级:** 中等

**缓解策略:** 开发者文档 + 代码审查

### 3.3 白名单维护疏漏

**问题:** 新增协议字符串时忘记更新白名单

**风险等级:** 中等

**缓解策略:** 定期审查 + CI 检查增强

## 4. 缓解策略设计

### 4.1 双层保护机制（per D-03, D-04）

**第一层: CI 检查（内置翻译）**

位置: scripts/check_protocol_strings.py

功能:
- 扫描 lib/i18n/**/*.json 翻译文件
- 检查翻译值是否在白名单中
- 发现违规时阻止 PR 合并

触发条件:
- PR 修改翻译文件
- 推送到 main 分支

**第二层: 运行时验证（外部翻译）**

位置: lib/i18n_core.py

功能:
- 加载外部翻译时验证
- 检查翻译值是否在白名单中
- 发现违规时拒绝加载，记录错误日志（per D-04）

触发时机:
- 应用启动时
- 加载 ~/.ccb/i18n/ 文件时

### 4.2 白名单维护流程

**新增协议字符串时:**
1. 在代码中使用常量（不通过 t() 翻译）
2. 添加到 .planning/protocol_whitelist.json 对应分类
3. 更新 total_count 和 last_updated
4. 提交白名单更新

**定期审查:**
- 频率: 每个 Phase 结束时
- 检查项: 完整性、废弃字符串、分类合理性

### 4.3 开发者教育

**文档化内容:**
- 协议字符串定义和识别方法
- 正确使用模式（常量 vs t() 函数）
- 常见错误示例

**代码审查清单:**
- [ ] 新增的 t() 调用不包含协议字符串
- [ ] 协议字符串使用常量定义
- [ ] 新增协议字符串已添加到白名单

## 5. 残留风险评估

### 5.1 白名单不完整

- **概率:** 中等
- **影响:** 高
- **缓解:** 定期审查 + 代码审查
- **可接受性:** 是（通过流程控制）

### 5.2 运行时验证性能影响

- **概率:** 低
- **影响:** 低
- **缓解:** 仅在加载外部翻译时验证，启动时一次性完成
- **可接受性:** 是（性能影响可忽略）

### 5.3 假阳性（正常翻译值与协议字符串冲突）

- **概率:** 极低
- **影响:** 中等
- **缓解:** 协议字符串使用特殊前缀（CCB_），降低冲突概率
- **可接受性:** 是（概率极低）

## 6. 实施建议

### 6.1 优先级

1. **P0（必须）:** 创建白名单文件
2. **P0（必须）:** 实现 CI 检查脚本
3. **P1（重要）:** 实现运行时验证
4. **P2（建议）:** 编写开发者文档

### 6.2 时间估算

- CI 检查脚本: 8 小时
- 运行时验证: 8 小时
- 开发者文档: 4 小时
- 测试验证: 4 小时
- **总计: 24 小时（3 天）**

### 6.3 验收标准

- [ ] 白名单文件包含所有 Phase 1 识别的协议字符串
- [ ] CI 检查脚本能检测翻译文件中的协议字符串
- [ ] 运行时验证能拒绝包含协议字符串的外部翻译
- [ ] 开发者文档清晰说明协议字符串使用规范

## 7. 结论

协议字符串误翻译是 i18n 改造的最高风险，但通过双层保护机制（CI 检查 + 运行时验证）可以有效缓解。残留风险主要来自白名单维护疏漏，可通过定期审查和代码审查流程控制。

**风险等级:** 高 → 低（缓解后）
**可接受性:** 是
**建议:** 优先实施双层保护机制
