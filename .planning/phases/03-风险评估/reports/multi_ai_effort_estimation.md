# 多 AI 集成工作量估算报告

**报告日期:** 2026-03-30  
**Phase:** 03-风险评估  
**需求:** RISK-05  
**依据文档:** `ccb_cli_backend_design_v3.md`、Phase 3 Context/Research

---

## 1. 执行摘要

本报告评估 GSD 在 CCB 环境中接入多 AI 协作能力的实施工作量。

锁定结论如下：

- **总工作量范围：40-60 小时**
- **中位估算：52 小时**
- **建议缓冲后口径：60 小时**
- **技术复杂度：中等**

复杂度被评为“中等”，原因不是底层协议本身复杂，而是需要把现有 CCB CLI 能力包装成稳定、可测试、可集成的后端抽象。

该工作的最小可用实现围绕 **CCBCLIBackend** 展开，不建议在当前阶段直接扩展到 MCP 或更重的抽象层。

---

## 2. 估算范围与边界

本报告覆盖的实现范围包括：

- `CCBCLIBackend` 的核心接口实现
- `TaskHandle` / `TaskResult` 模型
- `process_lock.py` 的集成与超时配置
- 单元测试与集成测试
- 文档与示例
- 可选的 GSD 对接工作

本报告不覆盖：

- 新建新的 daemon 协议
- 修改 CCB 底层会话架构
- MCP backend 的生产化实现
- 大规模 UI 层改造

因此，这里的 40-60 小时是“基于现有 CCB 能力包装”的工作量，而不是“重新设计多 AI 系统”的工作量。

---

## 3. 复杂度评估

### 3.1 复杂度等级

- **中等**

### 3.2 评为中等的原因

有利因素：

- 已有现成的 CCB CLI 命令可复用
- `ask` / `pend` / `ccb-ping` / `ccb-mounted` 已形成基本能力面
- `process_lock.py` 已提供跨平台文件锁基础
- `ccb_cli_backend_design_v3.md` 已冻结退出码映射和单任务约束

增加复杂度的因素：

- `poll()` 的状态映射必须严格正确
- Windows `subprocess` 行为需要单独验证
- 单 provider 单活跃任务约束必须文档化并可测试
- 文件锁超时与错误处理会直接影响调用可靠性

结论上，这不是“研发不确定性很高”的工作，但确实需要严谨工程化处理，不能简单按几条 shell 命令包装了事。

## 4. 工作量分解

### 4.1 总览

| 工作包 | 工时 |
|---|---:|
| CCBCLIBackend 实现 | 16 小时 |
| TaskHandle / TaskResult 模型 | 4 小时 |
| 文件锁集成 | 8 小时 |
| 测试覆盖 | 16 小时 |
| 文档编写 | 8 小时 |
| GSD 集成（可选） | 8 小时 |
| **合计（含可选项）** | **60 小时** |

如果暂不做 GSD 集成，则主估算为：

- **52 小时**

因此，推荐向外部汇报时采用：

- **40-60 小时范围**
- **52 小时中位值**

### 4.2 CCBCLIBackend 实现：16 小时

拆分如下：

- 核心类框架：8 小时
- 错误处理：4 小时
- Windows 兼容：4 小时

核心类实现应覆盖以下方法：

- `submit()`
- `poll()`
- `ping()`
- `list_providers()`

其中 `poll()` 是最关键点，因为它必须遵循 v3 设计中的退出码映射：

- `0 -> completed`
- `2 -> pending`
- `1 -> error`

### 4.3 TaskHandle / TaskResult 模型：4 小时

这部分工作量不大，但属于必要基础设施。

内容包括：

- 数据结构定义
- 字段约束确认
- 序列化/调试输出支持

如果这一层定义模糊，后面的 GSD 集成会反复返工。

### 4.4 文件锁集成：8 小时

拆分如下：

- `process_lock.py` 集成：4 小时
- 超时配置：2 小时
- 错误处理：2 小时

重点不在“有没有锁”，而在：

- 锁使用点是否覆盖正确
- 锁失败时返回什么语义
- 超时和重试是否足够可调

### 4.5 测试覆盖：16 小时

拆分如下：

- 单元测试：8 小时
- 集成测试：8 小时

建议优先覆盖：

- `submit/poll` 成功路径
- `EXIT_NO_REPLY` 场景
- 错误退出码和 stderr 映射
- 多 provider 并发
- 同 provider 单任务约束
- 文件锁超时与失败场景

### 4.6 文档编写：8 小时

拆分如下：

- API 文档：4 小时
- 使用示例：2 小时
- 最佳实践：2 小时

最重要的文档点有两个：

- 每个 provider 同时只能有一个活跃任务
- 不同 provider 可以并发，但同 provider 不能并发提交

### 4.7 GSD 集成（可选）：8 小时

这部分不建议默认算入“最小实现”。

内容包括：

- GSD 技能或执行器对接
- 调用结果映射到现有工作流
- 集成后的回归验证

如果当前目标只是验证 CCB 多 AI 后端抽象是否成立，这 8 小时可以后移。

## 5. 时间范围说明

### 5.1 为什么是 40-60 小时

低位估算接近 **40 小时** 的前提是：

- 不做 GSD 集成
- Windows 路径和 `subprocess` 没有意外问题
- 测试范围控制在核心接口

高位估算接近 **60 小时** 的前提是：

- 需要完整文档
- 需要做 GSD 对接
- 需要补齐跨平台和锁冲突边界测试

### 5.2 推荐对外口径

推荐统一表达为：

- “多 AI 集成的工作量约 **40-60 小时**，中位估算 **52 小时**，按缓冲后口径建议预留 **60 小时**。”

这样既满足计划要求，也便于后续排期。

## 6. 技术风险

### 6.1 subprocess 跨平台差异

- 概率：中
- 影响：中

风险点：

- Windows 下隐藏窗口、编码、超时行为
- 不同 shell 环境对命令调用的差异

缓解建议：

- 提前在 Windows 环境做最小链路验证
- 把 `subprocess.run()` 参数封装在一处

### 6.2 退出码映射错误

- 概率：低
- 影响：高

风险点：

- 把 `EXIT_NO_REPLY(2)` 误判成错误
- 错误场景只看 stdout，忽略 return code

缓解建议：

- 严格以 v3 设计文档为准
- 测试覆盖全部退出码

### 6.3 文件锁行为与性能

- 概率：低
- 影响：中

风险点：

- 锁获取失败时错误语义不清楚
- 超时过短导致误报
- 超时过长影响调用体验

缓解建议：

- 暴露锁超时配置
- 区分“等待中”和“失败”结果

### 6.4 单任务约束导致的使用误解

- 概率：中
- 影响：中到高

风险点：

- 用户或调用方误以为同 provider 可以并发多任务
- 轮询结果与提交顺序不一致时难以排查

缓解建议：

- 文档明确约束
- 运行时日志提示
- 测试里覆盖重复提交场景

## 7. 依赖关系

### 7.1 前置依赖

- CCB CLI 命令可用：`ask`、`pend`、`ccb-ping`、`ccb-mounted`
- `process_lock.py` 可复用
- `ccb_cli_backend_design_v3.md` 作为接口依据
- Python 3.10+ 运行环境

### 7.2 后置依赖

- GSD 执行器或技能层接入
- 使用文档更新
- 可能的示例或脚手架补充

### 7.3 关键技术依赖

真正的关键依赖不是第三方库，而是“已有 CLI 契约保持稳定”。
只要 `ask/pend/ping/mounted` 的行为不变，`CCBCLIBackend` 的工程实现风险就是可控的。

---

## 8. 与 i18n 改造的关系

### 8.1 关系判断

- **代码层面：弱耦合**
- **排期层面：中等耦合**

原因如下：

- `CCBCLIBackend` 主要处理任务提交、轮询和状态映射
- i18n 改造主要处理文案、翻译资产和协议保护
- 两者都接触 CCB CLI 输出和错误信息，因此在文档、日志和测试上存在交叉

### 8.2 协同点

协同点主要在三个地方：

1. 错误消息和用户提示最终都可能进入 i18n 体系
2. 协议字符串保护思想同样适用于多 AI 接口边界
3. 两项工作都依赖清晰的契约和测试，而不是“猜测式”集成

### 8.3 建议顺序

推荐顺序不是等待 536 小时 i18n 全量落地后再做多 AI 集成。

更合理的顺序是：

1. 先完成 i18n 原型中的协议保护验证
2. 并行或随后实现 `CCBCLIBackend`
3. 等接口稳定后，再考虑让多 AI 相关用户提示接入完整 i18n 体系

换句话说：

- **多 AI 集成不依赖 i18n 全量改造**
- **但应受益于 i18n 原型验证出来的协议保护规则**

推荐最小资源配置：

- **1 名 Python 开发者**

- **1 名 Python 开发者，预留 7.5 个工作日**

## 10. 结论

多 AI 集成以 **CCBCLIBackend** 为核心实现对象时，工作量边界较清晰：

- 范围估算：**40-60 小时**
- 中位估算：**52 小时**
- 缓冲口径：**60 小时**
- 技术复杂度：**中等**

它与 i18n 全量改造不存在强依赖，不需要等待 536 小时计划完成后才能启动。
但它应当继承 i18n 原型阶段对协议保护、契约稳定和测试方法的约束。

因此，当前最合理的执行建议是：

- 先确认 `CCBCLIBackend` 作为唯一优先后端
- 按 **52 小时** 编制实施计划
- 向排期和资源评估使用 **40-60 小时** 范围口径
