# 协议字符串误翻译风险评估报告

**评估日期:** 2026-03-30
**Phase:** 03-风险评估
**Plan:** 03-01
**需求:** RISK-01
**依据文档:** `03-CONTEXT.md`、`03-RESEARCH.md`、`protocol_protection_design.md`、`protocol_whitelist.json`

## 1. 执行摘要

- **严重程度:** 完全破坏（Critical）
- **结论:** 协议字符串一旦进入翻译层，最坏结果不是显示异常，而是 CCB/GSD 控制流失效。
- **核心原因:** 协议字符串承担的是命令、状态、路径、标记、环境变量和结构化键的“控制语义”，不是用户可读文案。
- **优先缓解策略:** 双层保护，先做 `CI 检查`，再做 `运行时验证`。
- **残留风险评级:** 低到中，主要来自白名单维护遗漏和 Phase 1 分类器偏差。

## 2. 数据基线

### 2.1 白名单实际计数

当前 `.planning/protocol_whitelist.json` 的实际白名单总数为 **300**。

| 分类 | 数量 | 说明 |
|---|---:|---|
| env_vars | 197 | 运行时环境变量、Provider 运行参数、目录和会话变量 |
| command_names | 6 | 关键命令字面量，如 `ask`、`cask`、`execute-phase` |
| completion_markers | 9 | 请求/响应/完成控制标记 |
| file_paths | 49 | 点前缀目录、扩展名、协议路径片段 |
| git_refs | 1 | 当前仅 `HEAD` |
| json_keys | 7 | 结构化计划与状态键 |
| config_keys | 31 | 常量名、枚举位、运行时配置键 |

### 2.2 来源说明

- Phase 1 `protocol` 桶提供了 **520** 条命中记录，对应 **287** 个唯一值。
- 本次修正额外补入了 **13** 个低熵控制字面量，来源仍是同一份 `classified.json`，但它们在 Phase 1 被放入了 `human` 桶。
- 补入项主要覆盖两类缺口：
- `command_names`：`ask`、`cask`、`config-get`、`execute-phase` 等。
- `json_keys`：`phase`、`plan`、`type`、`status`、`version` 等。

### 2.3 为什么这个偏差重要

- 如果白名单只依赖 `protocol` 桶，命令名和结构化键会出现保护盲区。
- 这类盲区不会降低翻译覆盖率，却会直接降低协议保护的有效性。
- 因此本次白名单不是机械复制分类结果，而是按“是否承载控制语义”重建。

## 3. 风险定级

### 3.1 严重程度

**完全破坏（Critical）**。

理由如下：

1. 协议字符串被翻译后，调用方和被调用方会失去共同语言。
2. 协议失配不会优雅退化，通常表现为无响应、路径失效、状态判断错误或任务卡死。
3. 这类错误常发生在进程边界、文件边界和配置边界，恢复成本高，定位成本也高。

### 3.2 影响范围

- CCB 守护进程与客户端之间的请求/响应协议
- 外部翻译覆盖加载流程
- 环境变量读取逻辑
- 规划文件与状态文件的结构化字段
- 带控制语义的路径、扩展名与完成标记

## 4. 影响场景

### 场景 1: 完成标记被翻译

- 原始协议值：`ask.response`
- 错误后果：被翻译为自然语言或本地化短语
- 直接影响：轮询端无法识别回复完成
- 结果：任务一直 pending，或者误判为超时/失败

### 场景 2: 请求/响应后缀被翻译

- 原始协议值：`.request`、`.response`、`.ping`、`.pong`
- 错误后果：请求文件和回复文件无法按约定命名
- 直接影响：生产者写入了“错误的协议文件名”，消费者监听“正确的协议文件名”
- 结果：跨进程通信断裂

### 场景 3: 环境变量名被翻译

- 原始协议值：`CCB_LANG`、`CCB_SESSION_ID`、`CODEX_SESSION_ID`
- 错误后果：读取不到变量或读到错误变量
- 直接影响：语言判断、会话恢复、Provider 路由失效
- 结果：回退逻辑异常、上下文丢失或行为漂移

### 场景 4: 结构化键被翻译

- 原始协议值：`phase`、`plan`、`type`、`status`
- 错误后果：解析器读不到预期键
- 直接影响：GSD 状态推进、计划解析、报告汇总失效
- 结果：状态文件合法但语义损坏，属于最难排查的一类错误

### 场景 5: 文件路径片段被翻译

- 原始协议值：`.planning`、`.json`、`.md`
- 错误后果：路径拼接变成不存在的目录或扩展名
- 直接影响：计划文件、状态文件、摘要文件读写失败
- 结果：自动化流程在 I/O 层面中断

## 5. 风险来源

### 5.1 外部翻译绕过 CI

- 对应用户决策：D-02、D-03、D-04
- 风险描述：`~/.ccb/i18n/` 不在版本控制中，CI 无法提前检查。
- 风险等级：高
- 发生概率：中
- 备注：这是最关键的单点缺口，也是必须增加运行时验证的原因。

### 5.2 开发者误把协议值送进 `t()`

- 风险描述：把命令名、标记、路径片段或 JSON 键当成可翻译文案处理。
- 风险等级：中
- 发生概率：中
- 典型模式：`command = t("...")`、`if t("...") == "..."`。

### 5.3 白名单维护遗漏

- 风险描述：新增协议字符串但没有同步更新白名单。
- 风险等级：中
- 发生概率：中
- 直接后果：CI 和运行时验证都不会拦住新字面量。

### 5.4 分类器偏差

- 风险描述：Phase 1 的自动分类把少量协议字面量归入了 `human`。
- 风险等级：中
- 发生概率：已发生
- 本次体现：13 个关键控制字面量需要人工补回白名单。

## 6. 双层保护设计

### 6.1 第一层: CI 检查

**目标:** 阻止内置翻译把协议字符串写进翻译值。

**输入:**

- `.planning/protocol_whitelist.json`
- `lib/i18n/**/*.json`

**检查动作:**

1. 加载白名单并展开为集合。
2. 扫描所有翻译文件的 value。
3. 只要 value 命中白名单，立即报错并阻止合并。

**适用范围:**

- 仓库内置翻译
- Pull Request
- 主分支推送

**优势:**

- 成本低
- 反馈早
- 可直接挂到 branch protection

**局限:**

- 不能覆盖用户本机的外部翻译文件

### 6.2 第二层: 运行时验证

**目标:** 补足外部翻译绕过 CI 的空洞。

**触发时机:**

- 应用启动
- 加载 `~/.ccb/i18n/` 目录时

**处理规则:**

1. 加载外部翻译。
2. 对每个翻译 value 执行白名单比对。
3. 发现命中时拒绝该外部翻译文件。
4. 记录错误日志。
5. 回退到内置翻译。

**与 D-03 / D-04 的一致性:**

- D-03 要求：运行时验证必须存在。
- D-04 要求：验证失败时拒绝加载并落日志。

### 6.3 两层机制为什么必须并存

- 只有 `CI 检查`：挡不住本地外部翻译。
- 只有 `运行时验证`：问题发现太晚，且没有 PR 门禁。
- 两层一起上：既有左移防线，也有最后兜底。

## 7. 白名单维护流程

### 7.1 新增协议字符串时

1. 先确认它是否承担控制语义，而不是用户文案。
2. 在代码中保持常量直出，不通过 `t()` 获取。
3. 把字面量加入 `.planning/protocol_whitelist.json` 的正确分类。
4. 如果是新命令或新结构化键，优先检查是否会落在分类器盲区。
5. 更新 `total_count` 和 `last_updated`。
6. 运行本地校验，确认分类非空、条目互斥、总数正确。

### 7.2 审查频率

- 每个风险评估或设计变更 Phase 结束时至少审查一次。
- 每次新增命令、完成标记、状态键或外部配置键时立即审查。

### 7.3 审查重点

- 是否新增了控制字面量但未入白名单
- 是否存在重复归类
- 是否把人类可读句子误放入白名单
- 是否有旧协议值已废弃但仍保留

## 8. 残留风险

### 8.1 白名单仍可能不完整

- 概率：中
- 影响：高
- 原因：Phase 1 分类器已经证明会漏掉低熵控制词
- 缓解：后续 CI 脚本除“翻译值检查”外，再增加“源码静态扫描”以发现新字面量

### 8.2 正常翻译值与协议值碰撞

- 概率：低
- 影响：中
- 原因：个别简短词如 `HEAD`、`JSON`、`IDLE` 通用性较高
- 缓解：对白名单中的通用短词进行重点复审；必要时补来源注释或来源清单

### 8.3 运行时日志不足

- 概率：中
- 影响：中
- 原因：如果拒绝加载时没有输出足够上下文，用户很难自查外部翻译问题
- 缓解：日志至少包含文件路径、命中的键和值、回退动作

## 9. 实施建议

### 9.1 P0

1. 基于当前白名单实现 `CI 检查` 脚本。
2. 在 `i18n_core` 加入 `运行时验证`。
3. 对外部翻译加载失败输出可诊断日志。

### 9.2 P1

1. 给协议字符串新增源码静态扫描。
2. 把 `command_names` 和 `json_keys` 的补录逻辑文档化。
3. 在开发者文档中明确“协议字符串永不翻译”的规则。

### 9.3 P2

1. 为通用短词建立来源清单。
2. 评估是否需要把白名单生成步骤半自动化。

## 11. 结论

协议字符串误翻译不是普通的本地化质量问题，而是协议完整性问题。

在当前代码库中，风险已经被量化为：

- **总保护面:** 300 个白名单字面量
- **最高敏感面:** 197 个环境变量/运行时变量
- **最脆弱缺口:** 13 个曾被自动分类遗漏的低熵控制字面量

综合 `03-CONTEXT.md` 的 D-01 到 D-04，本计划的正确方向是：

1. 先把白名单作为唯一事实源稳定下来。
2. 再用 `CI 检查` 封住内置翻译入口。
3. 最后用 `运行时验证` 封住外部翻译入口。

这样做之后，风险可以从“完全破坏且高暴露”降到“低到中、可监控、可审计”。
