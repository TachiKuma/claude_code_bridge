   - 文档目的：呈现 i18n 和多 AI 集成的完整技术方案
   - 范围：CCB（98 Python 文件）+ GSD（18 JS 文件）
   - 代码库基线：9,029 条人类可读文本，300 项协议字符串白名单

2. **i18n_core 架构设计**（80-100 行）
   综合自 i18n_core_design.md 和 Phase 4 原型验证结果：
   - **设计目标**：命名空间隔离（ccb.*/gsd.*）、回退链（当前语言 -> en -> 键名）、外部覆盖（~/.ccb/i18n/）
   - **核心 API**：`t(key, **kwargs)` 保持与现有 i18n.py 兼容，增加命名空间前缀
   - **翻译文件格式**：JSON，按 ccb/gsd/common 目录组织，key 使用 `ccb.{category}.{specific}` 命名规范
   - **语言检测**：CCB_LANG 环境变量优先，回退到系统 locale
   - **伪翻译支持**：xx 语言码，所有值添加 [<< >>] 标记和 x-padding
   - **原型验证结果**：56 条消息迁移至 en.json/zh.json/xx.json，11 个单元测试通过，向后兼容层（lib/i18n.py 的 _key_mapping）确保旧代码无需修改
   - 包含一个精简的类图或方法列表：
     ```
     class I18nCore:
       load_translations(domain, lang) -> None
       t(key, **kwargs) -> str
       _detect_language() -> str
       _load_json_file(path) -> dict
       _validate_no_protocol_strings() -> None
       _load_whitelist() -> set
     ```

3. **CCBCLIBackend 接口设计**（60-80 行）
   综合自 ccb_cli_backend_design_v3.md 和 Phase 4 原型结果：
   - **设计目标**：将 CCB CLI 命令包装为稳定的 Python API，供 GSD 或其他消费者使用
   - **4 个核心方法**：
     - `submit(provider, prompt, timeout=None) -> TaskHandle`
     - `poll(handle, timeout=None) -> TaskResult`
     - `ping(provider) -> bool`
     - `list_providers() -> list[str]`
   - **退出码映射**：EXIT_OK(0) -> completed, EXIT_NO_REPLY(2) -> pending, EXIT_ERROR(1) -> error
   - **TaskHandle/TaskResult**：dataclass，支持 to_dict()/to_json() 序列化，TaskResult 有 is_done/is_success 属性
   - **单任务约束**：每个 provider 同时只能有一个活跃任务（CCB 会话管理的固有限制）
   - **原型验证结果**：25 个 mock 测试通过，覆盖全部方法和错误路径（超时、FileNotFoundError、无效 JSON、未知退出码）
   - **Windows 兼容**：STARTF_USESHOWWINDOW、CREATE_NO_WINDOW 标志

4. **协议字符串保护机制**（40-50 行）
   综合自 protocol_protection_design.md 和 Phase 4 验证：
   - **双层保护架构**：CI 检查（阻止合并）+ 运行时验证（日志警告，回退到内置翻译）
   - **白名单**：300 项，7 个分类（env_vars 197、command_names 6、completion_markers 9、file_paths 49、git_refs 1、json_keys 7、config_keys 31）
   - **CI 检查脚本**：scripts/check_protocol_strings.py（108 行），加载白名单后扫描翻译文件值，命中则 exit 1
   - **运行时验证**：I18nCore._validate_no_protocol_strings()，在 load_translations() 时触发
   - **原型验证结果**：12 个测试通过，CI 对干净翻译 PASS，对违规翻译 FAIL 并检测 3 个违规项

5. **翻译文件组织结构**（30-40 行）
   综合自 translation_structure.md：
   - 目录结构：
     ```
     lib/i18n/
     ├── ccb/
     │   ├── en.json
     │   ├── zh.json
     │   └── xx.json
     ├── gsd/
     │   ├── en.json
     │   └── zh.json
     └── common/
         ├── en.json
         └── zh.json
     ```
   - 外部覆盖路径：`~/.ccb/i18n/{domain}/{lang}.json`
   - Key 命名规范：`{namespace}.{category}.{specific}`，如 `ccb.terminal.no_terminal_backend`

6. **实施路径**（40-50 行）
   基于架构设计和原型验证结果，描述从当前状态到完整实施的迁移路径：
   - **P0（必须）**：i18n_core 实现 + 协议保护 + 白名单/CI 校验 + 原型 50 条消息接入
   - **P1（重要）**：CCB 全量 98 文件替换 + GSD 18 文件替换 + 翻译文件系统化组织 + 测试覆盖
   - **P2（增强）**：自动化迁移工具 + 翻译记忆管理 + 更多格式化特性
   - 迁移策略：渐进式，按模块逐个迁移，保持向后兼容层（已验证的 _key_mapping 方案）
   - 向后兼容保证：lib/i18n.py 保留旧 API，通过 I18nCore 桥接
