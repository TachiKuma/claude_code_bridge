import sys
from pathlib import Path

def check_translations():

    with open(".planning/protocol_whitelist.json") as f:

核心逻辑：
```python
def check_translations():
    # 1. 加载白名单
    with open(".planning/protocol_whitelist.json") as f:
        whitelist_data = json.load(f)
        protocol_strings = set()
        for category, strings in whitelist_data["categories"].items():
            protocol_strings.update(strings)

    # 2. 扫描所有翻译文件
    errors = []
    for trans_file in Path("lib/i18n").rglob("*.json"):
        with open(trans_file, encoding="utf-8") as f:
            translations = json.load(f)

        # 3. 检查翻译值是否在白名单中
        for key, value in translations.items():
            if value in protocol_strings:
                errors.append({
                    "file": str(trans_file),
                    "key": key,
                    "value": value,
                    "reason": "协议字符串不应被翻译"
                })

    # 4. 报告错误
    if errors:
        print("❌ 协议字符串保护检查失败:")
        for error in errors:
            print(f"  {error['file']}: 键 '{error['key']}' 的值 '{error['value']}' 是协议字符串")
        sys.exit(1)

    print("✓ 协议字符串保护检查通过")
```

**4. 检查范围**
- 内置翻译：lib/i18n/**/*.json
- 外部翻译：~/.ccb/i18n/**/*.json（可选，仅在 CI 环境中检查内置翻译）
- 排除文件：.planning/protocol_whitelist.json 本身

**5. 错误检测逻辑**
检查条件：翻译文件中的值（value）是否在白名单中
- 正常情况：翻译值是人类可读文本，不在白名单中
- 错误情况：翻译值是协议字符串（如 "CCB_DONE"），在白名单中

示例：
```json
// lib/i18n/ccb/zh.json（错误示例）
{
  "ccb.env.lang": "CCB_LANG"  // ❌ 错误：CCB_LANG 是环境变量名，不应翻译
}

// lib/i18n/ccb/zh.json（正确示例）
{
  "ccb.env.lang_desc": "语言设置环境变量"  // ✓ 正确：描述性文本
}
```

**6. CI 集成方案（per D-14）**

**方案 A: GitHub Actions（推荐）**
```yaml
# .github/workflows/i18n-check.yml
name: i18n Protocol Check

on:
  pull_request:
    paths:
      - 'lib/i18n/**/*.json'
  push:
    branches:
      - main

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-python@v4
        with:
          python-version: '3.10'
      - name: Check protocol strings
        run: python scripts/check_protocol_strings.py
```

**方案 B: pre-commit hook（本地检查）**
```yaml
# .pre-commit-config.yaml
repos:
  - repo: local
    hooks:
      - id: check-protocol-strings
        name: Check protocol strings in translations
        entry: python scripts/check_protocol_strings.py
        language: system
        files: ^lib/i18n/.*\.json$
        pass_filenames: false
```

**推荐：同时使用两种方案**
- pre-commit hook：本地开发时快速反馈
- GitHub Actions：PR 合并前强制检查（per D-14）

**7. 阻止合并机制（per D-14）**
- GitHub Actions 检查失败时，PR 状态为 "failed"
- 配置分支保护规则：要求 "i18n Protocol Check" 通过才能合并
- 开发者必须修复错误后重新提交

**8. 错误修复流程**
当 CI 检查失败时：
1. 查看错误报告，识别问题键值对
2. 判断：
   - 如果是误报（值确实应该是协议字符串）→ 使用常量而非翻译
   - 如果是错误翻译 → 修改为人类可读文本
3. 重新提交，触发 CI 重新检查

示例修复：
```python
# 错误做法
print(t("ccb.command.ask"))  # 翻译为 "ask"

# 正确做法
ASK_COMMAND = "ask"  # 使用常量
print(t("ccb.command.ask_desc", cmd=ASK_COMMAND))  # 翻译为 "使用 {cmd} 命令发送消息"
```

**9. 白名单维护流程**
当新增协议字符串时：
1. 在代码中使用常量定义（不通过 t() 翻译）
2. 将常量值添加到 .planning/protocol_whitelist.json 对应分类
3. 更新 total_count 和 last_updated 字段
4. 提交白名单更新

**10. 性能考虑**
- 检查时间：预估 <1 秒（扫描 ~10 个 JSON 文件）
- 内存占用：<10MB（加载白名单和翻译文件）
- CI 影响：可忽略（相比其他 CI 步骤）

**11. 局限性和风险**
- 仅检查翻译文件的值，不检查键名
- 不检查代码中是否错误使用 t() 包装协议字符串
- 依赖白名单完整性（遗漏的协议字符串不会被检测）

**缓解措施：**
- 定期审查白名单完整性
- 代码审查时注意协议字符串使用
