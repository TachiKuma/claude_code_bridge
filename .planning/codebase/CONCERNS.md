# 代码库关注点

**分析日期:** 2026-03-28

## 技术债务

### 异步通信超时检测

**问题:** Gemini 和 Codex 适配器依赖空闲超时检测来判断请求完成，当提供商忘记发送 `CCB_DONE` 标记时。

**文件:**
- `lib/askd/adapters/gemini.py` (第 185-261 行)
- `lib/askd/adapters/codex.py` (第 155-277 行)

**影响:**
- Gemini 默认 15 秒空闲超时，Codex 默认 8 秒
- 如果提供商响应缓慢但未完成，可能误判为完成
- 环境变量 `CCB_GEMINI_IDLE_TIMEOUT` 和 `CCB_CASKD_IDLE_TIMEOUT` 可配置但文档不完整

**修复方案:**
- 增强提供商提示以确保 `CCB_DONE` 标记一致性
- 实现更智能的完成检测（例如，基于内容模式而非时间）
- 添加详细的超时配置文档

### 会话管理复杂性

**问题:** 会话管理跨越多个适配器和协议，存在状态不一致风险。

**文件:**
- `lib/askd/adapters/base.py` (会话键计算)
- `lib/askd/adapters/gemini.py` (第 48-98 行，会话消息读取)
- `lib/askd/adapters/opencode.py` (第 60-74 行，会话键计算)

**影响:**
- 2299 行会话相关代码分散在多个文件
- 会话键计算逻辑在多个适配器中重复
- 并发会话访问可能导致 JSON 解析失败（重试 10 次，每次 50ms 延迟）

**修复方案:**
- 统一会话管理层，消除重复逻辑
- 实现会话锁定机制防止并发冲突
- 添加会话验证和恢复机制

### 请求 ID 格式兼容性

**问题:** OpenCode 适配器需要支持旧的十六进制格式和新的时间戳格式请求 ID。

**文件:** `lib/askd/adapters/opencode.py` (req_id 正则表达式匹配)

**影响:**
- 格式转换逻辑可能不完整
- 旧会话与新请求的兼容性问题
- 调试困难，缺乏格式验证

**修复方案:**
- 实现统一的请求 ID 生成和验证
- 添加格式转换工具函数
- 记录所有格式转换操作

## 已知 Bug 和限制

### OpenCode 死锁问题（已修复但需监控）

**症状:** 第二个异步调用总是失败

**文件:** `lib/askd/adapters/opencode.py`

**触发:** 连续发送两个异步请求到 OpenCode

**当前缓解:** 会话 ID 固定已修复（v5.2.6）

**监控:** 需要持续测试以确保不会回归

### Gemini 完成标记遗漏

**症状:** Gemini 有时不发送 `CCB_DONE` 标记，导致请求挂起

**文件:** `lib/askd/adapters/gemini.py` (第 259-261 行)

**当前缓解:** 15 秒空闲超时自动检测完成

**风险:** 如果响应在 15 秒内仍在生成，会被误判为完成

## 安全考虑

### 环境变量和凭证管理

**风险:** 多个提供商需要 API 密钥和认证令牌

**文件:**
- `install.ps1` (第 1-80 行，环境配置)
- 各适配器中的环境变量读取

**当前缓解:**
- 凭证存储在系统环境或配置文件中
- 安装脚本处理 UTF-8 编码

**建议:**
- 实现凭证加密存储
- 添加凭证轮换机制
- 审计凭证访问日志

### 跨进程通信安全

**风险:** OpenCode 适配器使用跨进程序列化锁

**文件:** `lib/askd/adapters/opencode.py` (第 96-100 行)

**当前缓解:** 使用 `ProviderLock` 进行同步

**建议:**
- 验证锁定机制防止竞态条件
- 实现锁定超时防止死锁
- 添加锁定监控和诊断

## 性能瓶颈

### 会话 JSON 解析重试

**问题:** Gemini 适配器在读取会话消息时重试 10 次，每次延迟 50ms

**文件:** `lib/askd/adapters/gemini.py` (第 48-61 行)

**当前性能:** 最坏情况 500ms 延迟

**改进方案:**
- 实现异步文件锁定而非轮询
- 使用文件监视器检测完成
- 减少重试次数或增加延迟

### 多适配器工作池

**问题:** 每个提供商维护独立的工作池，可能导致资源浪费

**文件:** `lib/askd/daemon.py` (第 56-96 行)

**当前设计:** 按提供商分离工作池

**改进方案:**
- 实现共享工作池与提供商路由
- 添加工作池大小配置
- 监控工作池利用率

## 可维护性问题

### 适配器代码重复

**问题:** 多个适配器实现相似的超时和完成检测逻辑

**文件:**
- `lib/askd/adapters/claude.py`
- `lib/askd/adapters/codex.py`
- `lib/askd/adapters/gemini.py`
- `lib/askd/adapters/opencode.py`

**影响:** 1083 行错误处理代码分散在多个文件

**修复方案:**
- 提取通用超时检测到基类
- 创建完成检测策略接口
- 实现适配器模板方法模式

### 错误处理不一致

**问题:** 错误处理逻辑在各适配器中不一致

**文件:** 各适配器的 `handle_exception` 方法

**影响:** 难以调试跨提供商问题

**修复方案:**
- 标准化错误分类
- 实现统一的错误恢复策略
- 添加错误追踪和分析

### 日志记录分散

**问题:** 日志写入分散在多个模块，缺乏统一格式

**文件:** 各模块中的 `_write_log` 调用

**改进方案:**
- 实现集中式日志管理
- 标准化日志格式和级别
- 添加日志聚合和分析
