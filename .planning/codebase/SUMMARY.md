# 代码库映射总结

**映射日期:** 2026-03-28

## 项目概览

**Claude Code Bridge (CCB)** 是一个多 AI 提供商异步消息传递系统，通过守护进程架构实现统一的 CLI 接口，支持 Claude、Codex、Gemini、Droid 等多个 AI 提供商。

## 核心架构

**架构模式:** 守护进程驱动的多提供商异步消息系统

**关键特性:**
- 每个提供商独立的守护进程架构
- 基于 Socket 的 RPC 通信协议
- 终端后端抽象（WezTerm、tmux）
- 基于会话的状态管理
- 轻量级提示注入机制

**层次结构:**
1. **CLI 层** - 用户命令接口（`bin/ask`, `bin/cask` 等）
2. **守护进程层** - 长期运行的提供商服务器
3. **通信层** - 提供商特定的协议实现
4. **终端后端层** - 终端多路复用器操作抽象
5. **会话管理层** - 持久化和解析 AI 会话
6. **注册表层** - 窗格到提供商的映射追踪

## 技术栈

**核心技术:**
- Python 3.10+ (主要语言)
- Bash/PowerShell (安装脚本)

## 关键关注点

### 高优先级问题

1. **异步通信超时检测**
   - Gemini 和 Codex 依赖空闲超时判断完成
   - 可能误判缓慢但未完成的响应
   - 需要更智能的完成检测机制

2. **会话管理复杂性**
   - 2,299 行会话相关代码分散在多个文件
   - 并发访问可能导致 JSON 解析失败
   - 需要统一的会话管理层和锁定机制

3. **测试覆盖不足**
   - 关键路径缺少测试（会话并发、超时边界、锁定失败）
   - 需要集成测试和混沌工程测试

### 技术债务

1. **适配器代码重复**
   - 1,083 行错误处理代码分散在多个适配器
   - 需要提取通用逻辑到基类

2. **大型文件复杂度**
   - `transfer.py` (604 行)、`daemon.py` (472 行)
   - 需要拆分为更小的模块

3. **类型提示缺失**
   - 影响 IDE 支持和重构能力
   - 需要添加完整的类型注解

## 架构优势

1. **模块化设计** - 清晰的功能分离
2. **适配器模式** - 灵活支持多个 AI 提供商
3. **异步架构** - 非阻塞的请求处理
4. **终端抽象** - 支持多种终端后端
5. **会话持久化** - 可靠的状态管理

## 扩展性考虑

**当前限制:**
- 单一守护进程架构（所有提供商共享）
- 本地文件系统会话存储
- 未定义的容量上限

**扩展路径:**
- 多守护进程架构 + 负载均衡
- 远程会话存储支持
- 会话缓存层
- 守护进程池管理
