## 2. i18n_core 架构设计

### 2.1 设计目标

i18n_core 是 CCB 和 GSD 共享的国际化核心模块，设计目标如下：

1. **命名空间隔离** -- 使用 `ccb.*` 和 `gsd.*` 前缀避免翻译键冲突
2. **回退链** -- 当前语言 -> 英文 -> 键名（调试友好）
3. **外部覆盖** -- 支持用户自定义翻译（`~/.ccb/i18n/`）
4. **协议保护** -- 内置运行时验证，防止协议字符串被错误翻译
5. **向后兼容** -- 保持现有 `t(key, **kwargs)` API 契约

### 2.2 核心类设计

i18n_core 的核心是 `I18nCore` 类，提供翻译加载、语言检测和翻译查找功能：

```python
class I18nCore:
    """共享 i18n 核心模块

    提供命名空间隔离的翻译查找，支持外部翻译覆盖和运行时协议保护。
    """

    def __init__(self, namespace: str = "ccb"):
        """初始化 i18n 核心

        Args:
            namespace: 命名空间前缀（ccb, common）
        """
        self.namespace = namespace
        self.translations: Dict[str, str] = {}
        self.current_lang: Optional[str] = None
        self.logger = logging.getLogger(f"i18n.{namespace}")

    def load_translations(self) -> None:
        """加载翻译文件

        加载顺序：
        1. 内置翻译文件（lib/i18n/{namespace}/{lang}.json）
        2. 外部翻译文件（~/.ccb/i18n/{namespace}/{lang}.json）
        3. 外部翻译覆盖内置翻译（dict.update）
        4. 触发运行时协议字符串验证
        """

    def t(self, key: str, **kwargs) -> str:
        """翻译函数，保持与现有 i18n.py 兼容

        Args:
            key: 翻译键（如 "ccb.terminal.no_terminal_backend"）
            **kwargs: 格式化参数

        Returns:
            翻译后的消息，键缺失时返回键名本身（便于调试）
        """

    def _detect_language(self) -> str:
        """语言检测

        优先级：
        1. CCB_LANG 环境变量
        2. 系统 locale（LANG, LC_ALL）
        3. 默认英文
        """

    def _load_json_file(self, path: Path) -> dict:
        """加载 JSON 翻译文件，解析失败时返回空字典"""

    def _validate_no_protocol_strings(self) -> None:
        """运行时协议字符串验证，命中时记录日志警告（不抛异常）"""

    def _load_whitelist(self) -> set:
        """加载协议字符串白名单（.planning/protocol_whitelist.json）"""
```

### 2.3 翻译查找机制

翻译查找采用四级回退链：

```
1. 外部翻译文件（~/.ccb/i18n/{namespace}/{lang}.json）
   未找到
2. 内置翻译文件（lib/i18n/{namespace}/{lang}.json）
   未找到
3. 英文回退（如果当前语言非英文）
   未找到
4. 返回键名本身（便于调试，不中断运行）
```

**示例:**

| 场景 | 查找键 | 当前语言 | 结果 |
|------|--------|---------|------|
| 键存在于当前语言 | `ccb.error.no_terminal` | zh | "未检测到终端后端" |
| 键不存在，回退到英文 | `ccb.new.feature` | zh | "New feature" |
| 键完全不存在 | `ccb.unknown.key` | zh | "ccb.unknown.key"（返回键名） |

### 2.4 语言检测

语言检测基于 `CCB_LANG` 环境变量和系统 locale：

```python
def _detect_language(self) -> str:
    # 1. CCB_LANG 环境变量（最高优先级）
    ccb_lang = os.environ.get("CCB_LANG", "auto").lower()
    if ccb_lang in ("zh", "cn", "chinese"):
        return "zh"
    if ccb_lang in ("en", "english"):
        return "en"

    # 2. 系统 locale
    lang = os.environ.get("LANG", "") or os.environ.get("LC_ALL", "")
    if not lang:
        lang, _ = locale.getdefaultlocale()
        lang = lang or ""

    # 3. 默认英文
    return "zh" if lang.lower().startswith("zh") else "en"
```

支持的语言代码: `zh`（中文）、`en`（英文）、`xx`（伪翻译，用于测试 i18n 覆盖率）。

### 2.5 外部翻译覆盖

用户可在 `~/.ccb/i18n/` 目录下放置自定义翻译文件，覆盖内置翻译：

```json
// ~/.ccb/i18n/ccb/zh.json（用户自定义）
{
  "ccb.startup.backend_started": "后端已就绪"
}
```

仅需覆盖特定键，无需复制完整文件。外部翻译通过 `dict.update()` 覆盖内置翻译。

### 2.6 伪翻译支持

`xx` 语言码作为特殊伪翻译语言，所有翻译值添加 `[<< >>]` 标记和 x-padding：

```json
{
  "ccb.error.no_terminal": "[<< xxxx xxxx xx xxxxxxx xxxx (xxxxxx xx xxxx) >>]"
}
```

用途: 在开发阶段快速识别未经过 i18n 处理的硬编码字符串。

### 2.7 性能特征

| 指标 | 值 | 说明 |
|------|-----|------|
| 内存占用 | ~5KB | 56 条消息约 1.8KB，预估 CCB 全量 ~3-5KB |
| 查找速度 | 0.85 微秒 | 10,000 次测试平均值 |
| 冷启动时间 | <5ms | 加载 JSON + 构建字典 |

## 3. CCBCLIBackend 接口设计

### 3.1 设计目标

CCBCLIBackend 将 CCB CLI 命令（ask/pend/ccb-ping/ccb-mounted）包装为稳定的 Python API，供 GSD 或其他消费者使用：

- 提供异步任务提交和结果轮询的标准化接口
- 使用结构化数据传递（TaskHandle/TaskResult），避免解析控制台文本
- 统一的错误处理机制（error-as-value 模式）
- 跨平台兼容（Windows/Unix）

### 3.2 四个核心方法

```python
class CCBCLIBackend:
    """CCB CLI 包装接口，提供结构化的多 AI 协作能力

    约束：每个 provider 同时只能有一个活跃任务。
    """

    def submit(self, provider: str, prompt: str,
               context: Optional[Dict] = None) -> TaskHandle:
        """提交任务到 AI 提供商

        Args:
            provider: AI 提供商名称（codex, droid, gemini, claude）
            prompt: 任务描述文本
            context: 可选的上下文信息

        Returns:
            TaskHandle 对象，包含 provider 和 timestamp
        """

    def poll(self, handle: TaskHandle,
             timeout: Optional[float] = None) -> TaskResult:
        """轮询任务结果

        Args:
            handle: submit() 返回的任务句柄
            timeout: 可选的轮询超时时间

        Returns:
            TaskResult 对象，包含 status、output 和 error
        """

    def ping(self, provider: str) -> bool:
        """检查提供商连接状态

        Args:
            provider: AI 提供商名称

        Returns:
            True 表示连接正常，False 表示不可用
        """

    def list_providers(self) -> List[str]:
        """列出所有可用的 AI 提供商

        Returns:
            已挂载的提供商名称列表
        """
```

### 3.3 退出码映射

`poll()` 方法通过 CCB 退出码判断任务状态，映射关系如下：

| 退出码 | CCB 常量 | TaskResult.status | 说明 |
|--------|----------|-------------------|------|
| 0 | EXIT_OK | completed | 有回复，任务完成 |
| 2 | EXIT_NO_REPLY | pending | 无回复，仍在处理 |
| 1 | EXIT_ERROR | error | 命令执行错误 |
| 其他 | -- | error | 未知退出码 |

这个映射关系在 v3 设计中修正了早期版本将 EXIT_NO_REPLY(2) 误判为错误的 bug。

### 3.4 TaskHandle 和 TaskResult

```python
@dataclass
class TaskHandle:
    """任务句柄，用于跟踪提交的任务"""
    provider: str         # AI 提供商名称（同时作为任务标识）
    timestamp: float      # 提交时间戳（Unix 时间，秒）

    def to_dict(self) -> dict: ...
    def to_json(self) -> str: ...

@dataclass
class TaskResult:
    """任务结果，包含执行状态和输出"""
    provider: str                   # 对应的提供商名称
    status: str                     # 状态：pending, completed, error
    output: Optional[str] = None    # 成功时的输出内容
    error: Optional[str] = None     # 失败时的错误信息

    @property
    def is_done(self) -> bool: ...    # status != "pending"

    @property
    def is_success(self) -> bool: ... # status == "completed"

    def to_dict(self) -> dict: ...
    def to_json(self) -> str: ...
```

**关键设计决策:**
- 使用 provider 作为任务标识（而非客户端生成的 task_id），因为 CCB 每个 provider 同时只能有一个活跃任务
- 错误通过 TaskResult(status='error') 返回，不抛出异常（error-as-value 模式）
- 支持序列化（to_dict/to_json）便于日志记录和调试

> Source: Phase 2, task_models_design.md, ccb_cli_backend_design_v3.md

### 3.5 单任务约束

**每个 provider 同时只能有一个活跃任务**，这是 CCB 会话管理的固有限制。

正确用法 -- 不同 provider 可以并发：

```python
# 并发提交到不同 provider（安全）
handles = {
    "codex": backend.submit("codex", "Review code"),
    "droid": backend.submit("droid", "Review code"),
    "gemini": backend.submit("gemini", "Review code")
}
```

错误用法 -- 同一 provider 不能并发：

```python
handle1 = backend.submit("codex", "Task 1")
handle2 = backend.submit("codex", "Task 2")  # 会覆盖 Task 1
```

CCBCLIBackend 通过 ProviderLock（文件锁）串行化同一 provider 的请求。

### 3.6 错误处理策略

所有方法均遵循 error-as-value 模式，不抛出异常：

| 方法 | 正常返回 | 错误返回 |
|------|---------|---------|
| submit() | TaskHandle | 始终返回 TaskHandle，错误延迟到 poll() |
| poll() | TaskResult(status='completed') | TaskResult(status='error') |
| ping() | True | False |
| list_providers() | List[str] | []（空列表） |

### 3.7 Windows 兼容

CCBCLIBackend 在 Windows 平台使用特殊 subprocess 标志避免弹出控制台窗口：

```python
if sys.platform == "win32":
    startupinfo = subprocess.STARTUPINFO()
    startupinfo.dwFlags |= subprocess.STARTF_USESHOWWINDOW
    startupinfo.wShowWindow = subprocess.SW_HIDE
    kwargs["startupinfo"] = startupinfo
    kwargs["creationflags"] = subprocess.CREATE_NO_WINDOW
```

> Source: Phase 2, ccb_cli_backend_design_v3.md (Section 11)

CCBCLIBackend 原型已在 Phase 4 验证通过（PROTO-02, PROTO-04 SATISFIED）：

## 4. 协议字符串保护机制

### 4.1 问题定义

协议字符串是用于系统内部通信的标识符，包括环境变量名（如 `CCB_LANG`）、命令名称（如 `ask`）、完成标记（如 `CCB_DONE`）等。如果这些字符串被错误翻译，会导致跨进程通信失败。

### 4.2 双层保护架构

协议保护采用 CI 检查 + 运行时验证的双层机制：

| 层级 | 机制 | 触发时机 | 行为 |
|------|------|---------|------|
| 第一层 | CI 检查脚本 | 代码提交/PR 合并 | 检测到协议字符串时 exit 1，阻止合并 |
| 第二层 | 运行时验证 | I18nCore 加载翻译时 | 检测到协议字符串时记录日志警告，回退到内置翻译 |

### 4.3 白名单

协议字符串白名单存储在 `.planning/protocol_whitelist.json`，当前包含 300 项，分为 7 个分类：

| 分类 | 数量 | 示例 |
|------|------|------|
| env_vars（环境变量） | 197 | CCB_LANG, CCB_DONE, CCB_TERMINAL |
| file_paths（文件路径） | 49 | .planning, .md, .json, .gsd |
| config_keys（配置键） | 31 | gsd_state_version, status, progress |
| completion_markers（完成标记） | 9 | ask.response, CCB_DONE |
| command_names（命令名称） | 6 | ask, cpend, ccb-mounted |
| json_keys（JSON 键名） | 7 | phase, plan, type, auto |
| git_refs（Git 引用） | 1 | HEAD |

> Source: Phase 3, protocol_whitelist.json 和 protocol_mistranslation_risk.md

### 4.4 CI 检查脚本

CI 检查脚本 `scripts/check_protocol_strings.py`（108 行）实现以下逻辑：

1. 从 `.planning/protocol_whitelist.json` 加载 300 项白名单
2. 扫描 `lib/i18n/**/*.json` 中所有翻译文件的值
3. 如果翻译值命中白名单，报告违规并 exit 1
4. 支持 `--translations` 参数自定义扫描目录

**使用示例:**
```bash
# 默认扫描 lib/i18n/
python scripts/check_protocol_strings.py

# 自定义扫描目录
python scripts/check_protocol_strings.py --translations tests/fixtures/bad_translations
```

### 4.5 运行时验证

I18nCore 在 `load_translations()` 时自动调用 `_validate_no_protocol_strings()`，对比已加载的翻译值和白名单。命中时仅记录日志警告（不抛异常），确保应用不中断运行。

> Source: Phase 2, protocol_protection_design.md

### 4.6 原型验证结果

协议保护机制已在 Phase 4 验证通过（PROTO-03 SATISFIED）：

- **CI 脚本:** 加载 300 项白名单，对干净翻译文件 PASS，对违规翻译文件 FAIL 并检测到 3 个违规项
- **运行时验证:** `_validate_no_protocol_strings()` 集成到 I18nCore.load_translations()
- **测试:** 12 个测试覆盖白名单加载、正常翻译、违规检测、批量扫描和集成场景
- **测试夹具:** `tests/fixtures/bad_translations/zh_bad.json` 包含 3 个故意的协议字符串违规

> Source: Phase 4, 04-03-SUMMARY.md, 04-VERIFICATION.md

## 5. 翻译文件组织结构

### 5.1 目录结构

翻译文件按命名空间和语言代码组织：

```
lib/i18n/                          # 内置翻译根目录
+-- ccb/                           # CCB 命名空间
|   +-- en.json                    # 英文翻译
|   +-- zh.json                    # 中文翻译
|   +-- xx.json                    # 伪翻译（测试用）
+-- gsd/                           # GSD 命名空间（预留，当前冻结）
|   +-- en.json
|   +-- zh.json
+-- common/                        # 共享命名空间（预留扩展）
    +-- en.json
    +-- zh.json
```

外部翻译覆盖目录：

```
~/.ccb/i18n/                       # 用户自定义翻译根目录
+-- ccb/
|   +-- en.json
|   +-- zh.json
+-- common/
    +-- en.json
    +-- zh.json
```

### 5.2 Key 命名规范

翻译键采用 `{namespace}.{category}.{specific}` 三段式命名：

| 命名空间 | 分类 | 示例键 | 说明 |
|---------|------|--------|------|
| ccb | error | `ccb.error.no_terminal` | 终端检测错误 |
| ccb | startup | `ccb.startup.backend_started` | 启动成功消息 |
| ccb | command | `ccb.command.sending_to` | 命令执行提示 |
| ccb | connectivity | `ccb.connectivity.test_failed` | 连接测试失败 |
| common | error | `common.error.file_not_found` | 通用文件错误（预留） |

### 5.3 文件格式

翻译文件使用 JSON 格式，支持 Python `str.format()` 参数化：

```json
{
  "ccb.error.no_terminal": "未检测到终端后端 (WezTerm 或 tmux)",
  "ccb.startup.backend_started": "{provider} 已启动 ({terminal}: {pane_id})",
  "ccb.command.sending_to": "正在发送问题到 {provider}...",
  "ccb.connectivity.test_failed": "{provider} 连通性测试失败: {error}"
}
```

### 5.4 加载优先级

对于翻译键 `ccb.error.no_terminal`，加载顺序：

```
1. ~/.ccb/i18n/ccb/zh.json（用户自定义，最高优先级）
   未找到
2. lib/i18n/ccb/zh.json（内置翻译）
   未找到
3. lib/i18n/ccb/en.json（英文回退）
```

## 6. 实施路径

### 6.1 总体策略

采用渐进式迁移策略，按优先级分批推进。每次只改造少数模块，保持向后兼容层，确保不破坏现有功能。

### 6.2 P0 -- 必须完成（基础能力）

P0 是最小可用实施范围，确保 i18n 框架本身可用且安全：

| 工作项 | 预估工时 | 说明 |
|--------|---------|------|
| i18n_core 实现 | 24 小时 | 核心类、命名空间、外部覆盖、运行时验证 |
| 协议保护 + 白名单 | 16 小时 | CI 检查脚本、白名单维护流程 |
| 原型 50 条消息接入 | 8 小时 | 验证框架在实际场景中的可用性 |
| 基础测试 | 12 小时 | 单元测试 + 集成测试 |
| **P0 小计** | **60 小时** | |

**P0 交付标准:**
- I18nCore 可通过 t(key) 返回中英文翻译
- CI 检查可阻止协议字符串合并
- 50+ 条消息完成迁移，向后兼容层正常工作

### 6.3 P1 -- 重要（全量覆盖）

P1 将 i18n 覆盖扩展到全部代码库：

| 工作项 | 预估工时 | 说明 |
|--------|---------|------|
| CCB 98 文件全量替换 | 98 小时 | 按 1 小时/文件估算 |
| GSD 18 文件替换 | 36 小时 | 按 2 小时/文件估算 |
| 翻译文件系统化组织 | 300 小时 | 9029 条文本翻译、审校、组织 |
| 测试覆盖补齐 | 40 小时 | 单元测试 + 集成测试 |
| 文档更新 | 22 小时 | API 文档、开发规范 |
| **P1 小计** | **496 小时** | |

### 6.4 P2 -- 增强（提升效率）

P2 在 P1 基础上增加自动化和高级特性：

| 工作项 | 说明 |
|--------|------|
| 自动化迁移工具 | 从硬编码字符串自动提取和生成翻译键 |
| 翻译记忆管理 | 跨项目翻译复用、术语一致性维护 |
| 更多格式化特性 | 复数形式、性别、上下文相关翻译 |

### 6.5 迁移策略

迁移遵循以下原则：

1. **渐进式** -- 按模块逐个迁移，不一次性替换全量
2. **向后兼容** -- lib/i18n.py 保留旧 API，通过 `_key_mapping` 桥接到 I18nCore
3. **先保护后翻译** -- 先实现协议保护机制，再放开翻译文件编辑权限
4. **先原型后全量** -- 先完成 26 小时原型验证，再决定是否推进 536 小时完整实施

**向后兼容保证:**

现有代码调用 `t("no_terminal_backend")` 时，lib/i18n.py 通过 `_key_mapping` 字典（56 条映射）将其转换为 `ccb.terminal.no_terminal_backend`，再委托给 I18nCore.t()。旧代码无需任何修改即可在新的 i18n 框架下工作。
