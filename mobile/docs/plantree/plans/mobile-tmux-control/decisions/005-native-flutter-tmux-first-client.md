# Decision 005: Native Flutter Tmux-First Client

Date: 2026-06-18
Status: Proposed
Supersedes: [Decision 002](002-gateway-pwa-before-native-client.md)
Refines: [Decision 004](004-tmux-first-server-remote.md)

## Decision

Build the first serious CCB mobile/iPad client as a native cross-platform
Flutter app, while keeping the product tmux-first and server-remote.

Use ServerBox as the preferred native-client fork candidate if an AGPL mobile
component is acceptable. Use MuxPod as the primary tmux interaction reference
and as a source of Apache-2.0 ideas for pane navigation, polling, special keys,
deep links, reconnect, and multiline paste. Use Paseo and tmux-mobile as
protocol/gateway/terminal-stream references, not as the mobile app base.

The app should support a CCB-aware transport boundary:

- terminal plane: attach or interact with the CCB project tmux namespace;
- control plane: project/agent/status/lifecycle/content through CCB authority;
- pairing plane: QR/bootstrap with stable reconnect and device identity.

## Rationale

The user goal is not a generic browser terminal. The phone/iPad should be a
simple remote control for server-side CCB projects and panes.

The latest source review changed the base recommendation:

- tmux-mobile is MIT and useful for a server WebSocket terminal, but it is a
  web app and still needs native iOS/Android packaging, QR pairing, project
  model, and CCB status layers.
- ServerBox already has a mature Flutter iOS/Android terminal client, SSH
  session management, tmux attach/list/switch helpers, foreground service,
  Android notifications, and iOS Live Activity hooks.
- MuxPod is more tmux-focused than ServerBox and has better mobile pane UX, but
  it is less mature as a cross-platform product base and its README targets
  Android-first usage.
- Paseo already solves many product-level problems around QR pairing, daemon
  protocol, relay, terminal frames, and multi-device clients, but it is not a
  tmux-specific remote client and is AGPL.

## Consequences

- The first implementation should be a separate mobile app repository, not code
  copied into `ccb_source/lib/`.
- The app data model should be CCB project and agent first, not SSH host and
  arbitrary tmux session first.
- ServerBox's generic server-monitoring UI and destructive tmux/session
  actions should be removed or hidden for the CCB profile.
- ServerBox/MuxPod tmux command builders must be made socket-aware:
  `tmux -S <project_socket> ...`.
- A pure SSH/direct-tmux transport can be kept as a developer/fallback mode,
  but the product default should remain CCB-aware so project/agent status,
  completion reminders, Markdown content, and lifecycle actions do not depend
  on terminal scraping.
- The server gateway remains useful, but it becomes a CCB pairing/control
  sidecar and optional terminal proxy rather than the only mobile client.

## Validation Path

The decision is validated if a Flutter prototype can:

1. scan a QR code generated by the server-side CCB setup;
2. list CCB projects and favorites, not unrelated tmux sessions;
3. open a project terminal through the exact CCB tmux socket/session;
4. switch named agents and windows through CCB authority;
5. reconnect after network/app backgrounding without stopping CCB;
6. show completion/attention state without scraping terminal text;
7. render CCB Markdown, code blocks, tables, and math in a native reading view;
8. keep destructive tmux operations gated or absent by default.
